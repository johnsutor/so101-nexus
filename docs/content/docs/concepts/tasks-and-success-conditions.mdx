SO101-Nexus ships six task types that cover fundamental manipulation primitives.

## Task Overview

| Task | Objective | Max Steps |
| --- | --- | --- |
| PickLift | Grasp an object and lift it above a height threshold | 1024 |
| PickAndPlace | Pick up a cube and place it at a target location | 1024 |
| StackCube | Pick up cube A and stack it directly on top of cube B | 1024 |
| Touch | Bring the gripper to an object resting on the table | 512 |
| LookAt | Orient the end-effector to gaze at a target object | 256 |
| Move | Move the TCP a set distance in a cardinal direction | 256 |

## Success Conditions

Each task defines a binary success signal returned in the `info` dict.

**PickLift** succeeds when the object is grasped and lifted above the `lift_threshold` (default: 0.05 m).

**PickAndPlace** succeeds when the cube's XY distance to the goal is below `goal_thresh`, the cube is on the ground, and the robot is static.

**StackCube** succeeds when cube A's horizontal offset from cube B is within `sqrt(2) * cube_half_size + stack_alignment_margin` and its height above cube B is within `stack_alignment_margin` of exactly `2 * cube_half_size` (resting flush on cube B's top face), the robot is static, and cube A is no longer grasped -- releasing the cube is mandatory.

**Touch** succeeds when the TCP reaches within the target object's bounding radius plus `touch_margin` (default: 0.03 m), so it fires when the gripper reaches objects of any size.

**LookAt** succeeds when the target is within the wrist camera's field of view: the angle between the wrist-camera optical axis and the direction to the object is at most half the camera FOV (`fov_deg / 2`, or the live `cam_fovy / 2` when `fov_deg` is `None`). The dense reward shapes toward dead-center, rewarding a centered view.

**Move** succeeds when the TCP has traveled at least `target_distance` along the move direction (within `success_threshold`, default: 0.01 m), regardless of perpendicular drift.

## Reward Structure

Every step, the environment scores how well the robot is doing and hands back a
number (the reward) between about `-1` and `1`. Higher is better; `1.0` means
"you just finished the task." A policy is trained to rack up as much reward as
possible over an episode, so *how* the reward is computed shapes *what
behavior gets learned* -- this section explains the shape in plain terms.

The reward is a weighted sum of a few named pieces (`reaching`, `grasping`,
the task-specific objective, and a one-time `completion_bonus`), configurable
through [`RewardConfig`](/docs/api/configs) and required to sum to `1.0`. What
differs across tasks is *what each piece is measuring* and *when it pays out*:

### Simple tasks: reward reflects where you are right now

Touch, Move, and LookAt each have one thing to get right (reach a point, move
a fixed distance, point the camera at an object). For these, the reward each
step is just "how close are you to done right now" -- a smooth score that's
low far from the goal and rises toward `1.0` as the robot gets closer, plus a
one-time bonus the instant the goal is reached. There's nothing to game here:
since there's only one thing to accomplish and getting closer to it is always
the same direction as finishing, sitting still doesn't do anything special.

### Multi-step tasks: reward reflects *progress*, not position

PickLift, PickAndPlace, and StackCube involve several things that all have to happen in
order: reach the object, grasp it, then lift or carry-and-place (or carry-and-stack) it,
and end up still (PickAndPlace and StackCube additionally require releasing the object).
Here, the reward that isn't the one-time
completion bonus is not "how good does this look right now" -- it's "how much
*better* is this than the last step." Concretely: getting closer pays credit
once, while it's happening; once you stop getting closer, that piece of the
reward goes to exactly zero, whether you're standing still one meter away or
one millimeter away.

This matters because of a real trap a purely "how good does this look" reward
falls into: imagine the reward for lifting an object just measured how high it
currently is off the table. A robot arm that picks the object up and then just
*holds it there*, without ever finishing the task, would keep collecting that
"it's high up" reward every single step for the rest of the episode -- often
far more total reward than a robot that finishes quickly and ends the episode
early. The policy learns to freeze in a good-looking pose instead of finishing,
because standing still there simply pays better. Rewarding *progress* instead
of *position* closes that loophole: once you stop improving, the reward for
that piece stops too, so there's no benefit to parking anywhere short of
actually finishing. The formal name for this technique is *potential-based
reward shaping*; see `so101_nexus.rewards.potential_shaping` in the source for
the implementation if you're curious, but the plain-language version above is
the whole idea.

One consequence worth knowing: because the reward now reflects a *change*,
it can go slightly negative if you genuinely lose progress (for example,
dropping an object you'd already grasped), symmetric to the credit you'd earn
for making that same progress. This is intentional, not a bug -- it's what
makes "reward for progress" fair in both directions. Finishing the task always
pays the maximum reward (`1.0`) regardless of how the rest of the components
are behaving at that instant.

### Everything else

Distance-based pieces use a smooth curve (tanh shaping) rather than a hard
cutoff, so reward rises gradually as the robot gets closer rather than jumping
all at once. Optional penalties for jerky or high-effort motion
(`action_delta_penalty`, `energy_penalty`) can be turned on for smoother or
more efficient behavior; they're off by default.

See the [Configs API reference](/docs/api/configs) for all `RewardConfig`
parameters and default values.
