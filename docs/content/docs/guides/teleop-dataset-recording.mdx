---
title: Teleop Dataset Recording
description: Record demonstration datasets by teleoperating a simulated robot with a physical leader arm.
---

SO101-Nexus ships with a Gradio-based teleoperation recorder as part of the
core package. You control a simulated robot using a physical SO-100 or SO-101
leader arm, and demonstrations are written out as LeRobot v3 datasets ready
to push to the HuggingFace Hub.

## Installation

Teleop dependencies (`lerobot[feetech]`, `gradio`, `plotly`, `opencv-python`)
are packaged as an optional extra on `so101-nexus-core`. Install them with
your chosen backend:

```bash
# MuJoCo
uv sync --package so101-nexus-mujoco --extra teleop

# ManiSkill (note: ManiSkill and teleop must use separate environments)
uv sync --package so101-nexus-maniskill --extra teleop --prerelease=allow
```

With pip:

```bash
pip install "so101-nexus-core[teleop]" so101-nexus-mujoco
```

## Hardware requirements

- A physical SO-100 or SO-101 leader arm connected via USB.
- A Linux machine with the leader arm's serial port accessible.

## Finding the serial port

On Linux you may need to make the serial device writable before running the
port finder or launching teleop:

```bash
sudo chmod 666 /dev/ttyACM0
```

Use the LeRobot port discovery tool to identify the leader arm's port:

```bash
lerobot-find-port
```

## Launching the teleop recorder

Each backend exposes a `teleop` subcommand on a backend-specific CLI:

```bash
# MuJoCo
uv run so101-nexus-mujoco teleop --leader-port /dev/ttyACM0

# ManiSkill
uv run so101-nexus-maniskill teleop --leader-port /dev/ttyACM0
```

This opens a Gradio web UI in your browser. Most configuration happens in
the UI — the CLI only takes a handful of connection-level options.

### CLI arguments

| Argument | Type | Default | Description |
| --- | --- | --- | --- |
| `--leader-port` | `str` | `/dev/ttyACM0` | Serial port of the leader arm |
| `--leader-id` | `str` | `so101_leader` | Device identifier for the leader arm |
| `--wrist-roll-offset-deg` | `float` | `-90.0` | Wrist roll offset in degrees |

## Cameras and recorded fields

The recorder captures **both** the wrist camera and the overhead camera by
default, so you can decide after the fact which views you want in your
dataset without having to re-record.

Under **Advanced Settings → Dataset fields**, a progressively disclosed
accordion lets you choose which fields are persisted into the LeRobot
dataset:

- `observation.state` — always saved (disabled, cannot be unchecked)
- `action` — always saved (disabled, cannot be unchecked)
- `observation.images.wrist` — optional, on by default
- `observation.images.overhead` — optional, on by default
- `task` — optional, on by default

Deselecting an optional field removes it from both the declared dataset
feature schema and every recorded frame. `observation.state` and `action`
are forced on because a LeRobot dataset without them is not useful for
training.

### Camera resolution

Separate sliders for wrist and overhead camera width/height let you pick
the resolution captured at recording time. Both cameras are automatically
wired into the environment's observation config even if the task's default
config only includes one.

## Session flow

1. **Configure.** Pick the environment, robot type, episode count, FPS,
   camera resolutions, action space, countdown timer, and dataset fields.
2. **Record.** A countdown plays, then the simulation mirrors your leader
   arm in real time. The live wrist-camera feed shows in the UI.
3. **Review.** After each episode, the UI shows a video replay plus a
   joint-state plot.
4. **Approve or discard.** Accept the episode to write it into the dataset
   or discard and re-record.
5. **Push to Hub.** Once all episodes are done, push directly from the UI.

## Recorded dataset format

Datasets are saved in the LeRobot v3 format. Depending on which fields you
selected, recorded datasets may include:

- `observation.state` — joint positions (6D) at each timestep (always present)
- `action` — commanded joint positions or deltas (always present)
- `observation.images.wrist_cam` — wrist camera frames (optional)
- `observation.images.overhead_cam` — overhead camera frames (optional)
- `task` — language description of the task (optional)

`observation.state` contains the raw joint positions from the leader arm,
not the simulator's internal state vector, so datasets stay compatible with
sim-to-real transfer and vision-based policy training.
